## 检查清单的目的："需求的单元测试"

**核心概念**：检查清单是**需求编写的单元测试** - 它们验证特定领域需求的质量、清晰度和完整性。

**不是用于验证/测试实现**：

- ❌ 不是"验证按钮点击是否正确"
- ❌ 不是"测试错误处理是否有效"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否符合规格说明

**用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层次要求？"（完整性）
- ✅ "'突出显示'是否用具体的尺寸/定位进行了量化？"（清晰度）
- ✅ "所有交互元素的悬停状态要求是否一致？"（一致性）
- ✅ "是否为键盘导航定义了无障碍要求？"（覆盖度）
- ✅ "规格说明是否定义了 logo 图片加载失败时的处理方式？"（边界情况）

**比喻**：如果你的规格说明是用英语写的代码，那么检查清单就是它的单元测试套件。你正在测试需求是否编写得当、完整、无歧义、可供实现 - 而不是测试实现是否有效。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果非空）。

## 执行步骤

1. **设置**：从仓库根目录运行 `{SCRIPT}` 并解析 JSON 以获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须是绝对路径。
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **澄清意图（动态）**：生成最多三个初始的上下文澄清问题（不使用预设目录）。它们必须：
   - 从用户的措辞 + 从规格说明/计划/任务中提取的信号生成
   - 只询问会实质性改变检查清单内容的信息
   - 如果在 `$ARGUMENTS` 中已经明确，则单独跳过
   - 优先考虑精确性而非广度

   生成算法：
   1. 提取信号：功能领域关键词（如 auth、latency、UX、API）、风险指标（"关键"、"必须"、"合规"）、利益相关者提示（"QA"、"评审"、"安全团队"）、以及明确的交付物（"a11y"、"回滚"、"契约"）。
   2. 将信号聚类为候选关注领域（最多 4 个），按相关性排序。
   3. 识别可能的受众和时机（作者、评审者、QA、发布），如果未明确说明。
   4. 检测缺失的维度：范围广度、深度/严格性、风险重点、排除边界、可衡量的验收标准。
   5. 从以下原型中选择问题：
      - 范围细化（如，"这是否应该包括与 X 和 Y 的集成触点，还是仅限于本地模块的正确性？"）
      - 风险优先级（如，"这些潜在风险领域中，哪些应该有强制性的门控检查？"）
      - 深度校准（如，"这是轻量级的提交前检查清单还是正式的发布门控？"）
      - 受众定位（如，"这将仅供作者使用还是在 PR 评审期间供同行使用？"）
      - 边界排除（如，"本次是否应该明确排除性能调优项目？"）
      - 场景类别缺口（如，"未检测到恢复流程——回滚/部分失败路径是否在范围内？"）

   问题格式规则：
   - 如果提供选项，生成一个紧凑的表格，包含列：选项 | 候选项 | 重要性
   - 最多限制 A-E 个选项；如果自由形式的答案更清晰，则省略表格
   - 永远不要求用户重复他们已经说过的内容
   - 避免推测性类别（不要臆造）。如果不确定，明确询问："确认 X 是否在范围内。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：如果与代码相关则为评审者（PR）；否则为作者
   - 关注点：相关性最高的前 2 个聚类

   输出问题（标记为 Q1/Q2/Q3）。收到答案后：如果 ≥2 个场景类别（备选/异常/恢复/非功能领域）仍不清楚，你可以再问最多两个有针对性的后续问题（Q4/Q5），每个都附一行理由（如，"未解决的恢复路径风险"）。总问题数不要超过五个。如果用户明确拒绝更多问题，则跳过升级。

3. **理解用户请求**：结合 `$ARGUMENTS` + 澄清答案：
   - 推导检查清单主题（如，安全、评审、部署、用户体验）
   - 整合用户提到的明确必备项目
   - 将关注点选择映射到类别框架
   - 从规格说明/计划/任务中推断任何缺失的上下文（不要臆造）

4. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围
   - plan.md（如果存在）：技术细节、依赖项
   - tasks.md（如果存在）：实现任务

   **上下文加载策略**：
   - 只加载与当前关注领域相关的必要部分（避免全文件导入）
   - 优先将长章节总结为简洁的场景/需求要点
   - 使用渐进式披露：仅在检测到缺口时添加后续检索
   - 如果源文档较大，生成临时摘要项而不是嵌入原始文本

5. **生成检查清单** - 创建"需求的单元测试"：
   - 如果 `FEATURE_DIR/checklists/` 目录不存在则创建
   - 生成唯一的检查清单文件名：
     - 使用简短、描述性的名称，基于领域（如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[领域].md`
     - 如果文件存在，追加到现有文件
   - 从 CHK001 开始顺序编号项目
   - 每次 `/speckit.checklist` 运行都会创建一个新文件（永不覆盖现有检查清单）

   **核心原则 - 测试需求，而非实现**：
   每个检查清单项目必须评估需求本身的：
   - **完整性**：是否存在所有必要的需求？
   - **清晰度**：需求是否无歧义且具体？
   - **一致性**：需求之间是否一致？
   - **可衡量性**：需求是否可以客观验证？
   - **覆盖度**：是否涵盖了所有场景/边界情况？

   **类别结构** - 按需求质量维度分组：
   - **需求完整性**（是否记录了所有必要的需求？）
   - **需求清晰度**（需求是否具体且无歧义？）
   - **需求一致性**（需求是否一致无冲突？）
   - **验收标准质量**（成功标准是否可衡量？）
   - **场景覆盖**（是否涵盖了所有流程/情况？）
   - **边界情况覆盖**（是否定义了边界条件？）
   - **非功能性需求**（性能、安全、无障碍等是否已指定？）
   - **依赖项与假设**（是否已记录并验证？）
   - **歧义与冲突**（哪些需要澄清？）

   **如何编写检查清单项目 - "需求的单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证着陆页显示 3 个剧集卡片"
   - "测试桌面端悬停状态是否有效"
   - "确认点击 logo 导航到首页"

   ✅ **正确**（测试需求质量）：
   - "是否指定了精选剧集的确切数量和布局？" [完整性]
   - "'突出显示'是否用具体的尺寸/定位进行了量化？" [清晰度]
   - "所有交互元素的悬停状态要求是否一致？" [一致性]
   - "是否为所有交互式 UI 定义了键盘导航要求？" [覆盖度]
   - "是否指定了 logo 图片加载失败时的后备行为？" [边界情况]
   - "是否为异步剧集数据定义了加载状态？" [完整性]
   - "规格说明是否定义了竞争 UI 元素的视觉层次？" [清晰度]

   **项目结构**：
   每个项目应遵循以下模式：
   - 询问需求质量的问题形式
   - 关注规格说明/计划中写了什么（或没写什么）
   - 在方括号中包含质量维度 [完整性/清晰度/一致性/等]
   - 检查现有需求时引用规格说明章节 `[规格说明 §X.Y]`
   - 检查缺失需求时使用 `[缺口]` 标记

   **按质量维度的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理要求？ [缺口]"
   - "是否为所有交互元素指定了无障碍要求？ [完整性]"
   - "是否为响应式布局定义了移动端断点要求？ [缺口]"

   清晰度：
   - "'快速加载'是否用具体的时间阈值进行了量化？ [清晰度，规格说明 §NFR-2]"
   - "'相关剧集'的选择标准是否已明确定义？ [清晰度，规格说明 §FR-5]"
   - "'突出'是否用可衡量的视觉属性定义？ [歧义，规格说明 §FR-4]"

   一致性：
   - "所有页面的导航要求是否一致？ [一致性，规格说明 §FR-10]"
   - "着陆页和详情页之间的卡片组件要求是否一致？ [一致性]"

   覆盖度：
   - "是否为零状态场景（无剧集）定义了要求？ [覆盖度，边界情况]"
   - "是否涵盖了并发用户交互场景？ [覆盖度，缺口]"
   - "是否为部分数据加载失败指定了要求？ [覆盖度，异常流程]"

   可衡量性：
   - "视觉层次要求是否可衡量/可测试？ [验收标准，规格说明 §FR-1]"
   - "'平衡的视觉权重'是否可以客观验证？ [可衡量性，规格说明 §FR-2]"

   **场景分类与覆盖**（需求质量关注点）：
   - 检查是否存在以下场景的需求：主要、备选、异常/错误、恢复、非功能性场景
   - 对于每个场景类别，询问："[场景类型] 需求是否完整、清晰、一致？"
   - 如果场景类别缺失："[场景类型] 需求是故意排除还是遗漏的？ [缺口]"
   - 当发生状态变更时包含弹性/回滚："是否为迁移失败定义了回滚要求？ [缺口]"

   **可追溯性要求**：
   - 最低要求：≥80% 的项目必须包含至少一个可追溯性引用
   - 每个项目应引用：规格说明章节 `[规格说明 §X.Y]`，或使用标记：`[缺口]`、`[歧义]`、`[冲突]`、`[假设]`
   - 如果不存在 ID 体系："是否建立了需求和验收标准的 ID 方案？ [可追溯性]"

   **发现并解决问题**（需求质量问题）：
   询问关于需求本身的问题：
   - 歧义："'快速'这个术语是否用具体指标进行了量化？ [歧义，规格说明 §NFR-1]"
   - 冲突："§FR-10 和 §FR-10a 之间的导航要求是否冲突？ [冲突]"
   - 假设："'播客 API 始终可用'的假设是否已验证？ [假设]"
   - 依赖项："外部播客 API 要求是否已记录？ [依赖项，缺口]"
   - 缺失定义："'视觉层次'是否用可衡量的标准定义？ [缺口]"

   **内容整合**：
   - 软性上限：如果原始候选项目 > 40，按风险/影响优先排序
   - 合并检查相同需求方面的近似重复项
   - 如果 >5 个低影响的边界情况，创建一个项目："需求中是否涵盖了边界情况 X、Y、Z？ [覆盖度]"

   **🚫 绝对禁止** - 这些会使其成为实现测试，而非需求测试：
   - ❌ 任何以"验证"、"测试"、"确认"、"检查" + 实现行为开头的项目
   - ❌ 涉及代码执行、用户操作、系统行为的引用
   - ❌ "正确显示"、"正常工作"、"按预期运行"
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 流程
   - ❌ 实现细节（框架、API、算法）

   **✅ 必需模式** - 这些测试需求质量：
   - ✅ "是否为 [场景] 定义/指定/记录了 [需求类型]？"
   - ✅ "[模糊术语] 是否用具体标准进行了量化/澄清？"
   - ✅ "[章节 A] 和 [章节 B] 之间的需求是否一致？"
   - ✅ "[需求] 是否可以客观衡量/验证？"
   - ✅ "需求中是否涵盖了 [边界情况/场景]？"
   - ✅ "规格说明是否定义了 [缺失的方面]？"

6. **结构参考**：按照 `templates/checklist-template.md` 中的规范模板生成检查清单，包括标题、元数据部分、类别标题和 ID 格式。如果模板不可用，使用：H1 标题、目的/创建元数据行、`##` 类别章节，包含 `- [ ] CHK### <需求项目>` 行，ID 从 CHK001 开始全局递增。

7. **报告**：输出已创建检查清单的完整路径、项目数量，并提醒用户每次运行都会创建新文件。总结：
   - 选择的关注领域
   - 深度级别
   - 角色/时机
   - 纳入的任何用户明确指定的必备项目

**重要**：每次 `/speckit.checklist` 命令调用都会使用简短、描述性的名称创建检查清单文件，除非文件已存在。这允许：

- 不同类型的多个检查清单（如 `ux.md`、`test.md`、`security.md`）
- 简单、易记的文件名，表明检查清单的目的
- 在 `checklists/` 文件夹中轻松识别和导航

为避免混乱，请使用描述性类型，并在完成后清理过时的检查清单。

## 检查清单类型示例与样本项目

**UX 需求质量：** `ux.md`

样本项目（测试需求，而非实现）：

- "是否用可衡量的标准定义了视觉层次要求？ [清晰度，规格说明 §FR-1]"
- "是否明确指定了 UI 元素的数量和位置？ [完整性，规格说明 §FR-1]"
- "交互状态要求（悬停、聚焦、激活）是否一致定义？ [一致性]"
- "是否为所有交互元素指定了无障碍要求？ [覆盖度，缺口]"
- "是否定义了图片加载失败时的后备行为？ [边界情况，缺口]"
- "'突出显示'是否可以客观衡量？ [可衡量性，规格说明 §FR-4]"

**API 需求质量：** `api.md`

样本项目：

- "是否为所有失败场景指定了错误响应格式？ [完整性]"
- "是否用具体阈值量化了限流要求？ [清晰度]"
- "所有端点的认证要求是否一致？ [一致性]"
- "是否为外部依赖定义了重试/超时要求？ [覆盖度，缺口]"
- "版本策略是否记录在需求中？ [缺口]"

**性能需求质量：** `performance.md`

样本项目：

- "是否用具体指标量化了性能要求？ [清晰度]"
- "是否为所有关键用户旅程定义了性能目标？ [覆盖度]"
- "是否指定了不同负载条件下的性能要求？ [完整性]"
- "性能要求是否可以客观衡量？ [可衡量性]"
- "是否为高负载场景定义了降级要求？ [边界情况，缺口]"

**安全需求质量：** `security.md`

样本项目：

- "是否为所有受保护资源指定了认证要求？ [覆盖度]"
- "是否为敏感信息定义了数据保护要求？ [完整性]"
- "威胁模型是否已记录且需求与之对齐？ [可追溯性]"
- "安全要求是否与合规义务一致？ [一致性]"
- "是否定义了安全失败/违规响应要求？ [缺口，异常流程]"

## 反面示例：不应做什么

**❌ 错误 - 这些测试实现，而非需求：**

```markdown
- [ ] CHK001 - 验证着陆页显示 3 个剧集卡片 [规格说明 §FR-001]
- [ ] CHK002 - 测试桌面端悬停状态是否正常工作 [规格说明 §FR-003]
- [ ] CHK003 - 确认点击 logo 导航到首页 [规格说明 §FR-010]
- [ ] CHK004 - 检查相关剧集部分显示 3-5 个项目 [规格说明 §FR-005]
```

**✅ 正确 - 这些测试需求质量：**

```markdown
- [ ] CHK001 - 是否明确指定了精选剧集的数量和布局？ [完整性，规格说明 §FR-001]
- [ ] CHK002 - 所有交互元素的悬停状态要求是否一致定义？ [一致性，规格说明 §FR-003]
- [ ] CHK003 - 所有可点击品牌元素的导航要求是否清晰？ [清晰度，规格说明 §FR-010]
- [ ] CHK004 - 相关剧集的选择标准是否已记录？ [缺口，规格说明 §FR-005]
- [ ] CHK005 - 是否为异步剧集数据定义了加载状态要求？ [缺口]
- [ ] CHK006 - "视觉层次"要求是否可以客观衡量？ [可衡量性，规格说明 §FR-001]
```

**关键区别：**

- 错误：测试系统是否正常工作
- 正确：测试需求是否编写正确
- 错误：行为验证
- 正确：需求质量验证
- 错误："它是否执行 X？"
- 正确："X 是否清晰地指定了？"
