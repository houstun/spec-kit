## 概述

你正在更新位于 `/memory/constitution.md` 的项目宪法。该文件是一个包含方括号占位符标记的模板（例如 `[PROJECT_NAME]`、`[PRINCIPLE_1_NAME]`）。你的工作是 (a) 收集/推导具体值，(b) 精确填充模板，(c) 将任何修订传播到依赖工件。

按照以下执行流程：

1. 加载位于 `/memory/constitution.md` 的现有宪法模板。
   - 识别所有形如 `[ALL_CAPS_IDENTIFIER]` 的占位符标记。
   **重要**：用户可能需要比模板中使用的更少或更多的原则。如果指定了数量，请遵循——遵循通用模板。你将相应更新文档。

2. 收集/推导占位符的值：
   - 如果用户输入（对话）提供了值，使用它。
   - 否则从现有仓库上下文（README、文档、嵌入的先前宪法版本）推断。
   - 对于治理日期：`RATIFICATION_DATE` 是原始采纳日期（如果未知，询问或标记 TODO），`LAST_AMENDED_DATE` 是今天（如果有更改），否则保持之前的日期。
   - `CONSTITUTION_VERSION` 必须根据语义版本控制规则递增：
     - MAJOR：向后不兼容的治理/原则删除或重新定义。
     - MINOR：新增原则/章节或实质性扩展指导。
     - PATCH：澄清、措辞、拼写修正、非语义优化。
   - 如果版本升级类型不明确，在最终确定前提出推理。

3. 起草更新后的宪法内容：
   - 用具体文本替换每个占位符（不留任何带括号的标记，除非是项目选择暂时不定义的模板槽位——明确说明保留的理由）。
   - 保留标题层次结构，替换后可以删除注释，除非它们仍然提供澄清指导。
   - 确保每个原则部分：简洁的名称行、捕获不可协商规则的段落（或项目符号列表）、如果不明显则明确理由。
   - 确保治理部分列出修订程序、版本控制政策和合规审查期望。

4. 一致性传播检查清单（将先前的检查清单转换为主动验证）：
   - 读取 `/templates/plan-template.md` 并确保任何"宪法检查"或规则与更新后的原则一致。
   - 读取 `/templates/spec-template.md` 检查范围/需求一致性——如果宪法添加/删除了强制章节或约束，则更新。
   - 读取 `/templates/tasks-template.md` 并确保任务分类反映新的或删除的原则驱动的任务类型（例如，可观测性、版本控制、测试规范）。
   - 读取 `/templates/commands/*.md` 中的每个命令文件（包括此文件），验证没有过时的引用（仅限于 CLAUDE 等代理特定名称），当需要通用指导时。
   - 读取任何运行时指导文档（例如 `README.md`、`docs/quickstart.md` 或代理特定指导文件，如果存在）。更新对已更改原则的引用。

5. 生成同步影响报告（更新后作为 HTML 注释添加到宪法文件顶部）：
   - 版本变更：旧 → 新
   - 修改的原则列表（如果重命名则为旧标题 → 新标题）
   - 添加的章节
   - 删除的章节
   - 需要更新的模板（✅ 已更新 / ⚠ 待处理）及文件路径
   - 后续 TODO（如果有任何占位符被有意推迟）。

6. 最终输出前的验证：
   - 没有剩余的未解释括号标记。
   - 版本行与报告匹配。
   - 日期为 ISO 格式 YYYY-MM-DD。
   - 原则是声明性的、可测试的，没有模糊语言（"应该" → 在适当时替换为 MUST/SHOULD 理由）。

7. 将完成的宪法写回 `/memory/constitution.md`（覆盖）。

8. 向用户输出最终摘要：
   - 新版本和升级理由。
   - 标记为需要手动跟进的任何文件。
   - 建议的提交消息（例如 `docs: 将宪法修订为 vX.Y.Z（原则添加 + 治理更新）`）。

格式和样式要求：

- 完全按照模板中的 Markdown 标题使用（不要降级/升级级别）。
- 包装较长的理由行以保持可读性（理想情况下 <100 字符），但不要强制使用尴尬的换行。
- 在章节之间保持单个空行。
- 避免尾随空格。

如果用户提供部分更新（例如，仅一个原则修订），仍执行验证和版本决策步骤。

如果关键信息缺失（例如，批准日期确实未知），插入 `TODO(<FIELD_NAME>): explanation` 并在同步影响报告的推迟项目下包含。

不要创建新模板；始终操作现有的 `/memory/constitution.md` 文件。
